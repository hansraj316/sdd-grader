## User Scenarios & Testing *(mandatory)*

### Primary User Story

As a data analyst, I want to ask questions about Polymarket markets in natural
language from the AI assistant I already use, so that I can get market insights
without writing code.

**Acceptance Scenarios:**

1. **Given** the server is connected to the analyst's assistant, **When** the
   analyst asks "What are the top 5 markets by volume?", **Then** the answer
   contains a markdown table with exactly 5 rows ordered by 24-hour trading
   volume, each row showing the market question, its volume, and its current
   outcome prices.
2. **Given** market data exists for a named market, **When** the analyst asks
   for that market's details, **Then** the answer presents the market question,
   end date, outcome prices, and volume, and every value matches the Polymarket
   data source as of the answer's stated timestamp.
3. **Given** the analyst asks a question unrelated to Polymarket data (e.g.
   about the weather), **When** the question reaches the system, **Then** the
   answer states that only Polymarket market questions are supported and lists
   the kinds of questions it can answer.
4. **Given** the Polymarket data source is unreachable, **When** the analyst
   asks any data question, **Then** within 5 seconds the answer names the
   unavailable dependency and suggests retrying, with no fabricated or
   silently substituted data.
5. **Given** the analyst asks an ambiguous question such as "show me the top
   markets" (no metric named), **When** the system answers, **Then** the answer
   either asks one clarifying question or states the assumption it applied
   (default: 24-hour volume).

### Edge Cases

- **Polymarket data source unavailable** → the system reports the failure and
  the failing dependency within 5 seconds (FR-006, Scenario 4).
- **Ambiguous natural-language question** → the system asks one clarifying
  question or answers under a stated default assumption (FR-008, Scenario 5).
- **Question outside Polymarket data** → the system declines and states its
  supported scope (FR-007, Scenario 3).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST expose Polymarket market search and analysis as
  tools that any MCP-capable assistant can call, so the analyst's existing
  assistant serves as the natural-language interface. No bespoke chat window or
  command-line interface is in scope.
- **FR-002**: The system MUST retrieve current market, event, and trade data
  from the existing Polymarket data source.
- **FR-003**: The system MUST answer analysis questions deterministically from
  retrieved data: top-N ranking by a named metric, filtering by category, tag,
  date range, or open/closed status, and summary statistics (totals, averages,
  counts).
- **FR-004**: The system MUST return tabular results as markdown tables.
- **FR-005**: The system MUST include in every data answer the units of each
  figure and the timestamp the underlying data was fetched.
- **FR-006**: When the Polymarket data source is unavailable or returns an
  error, the system MUST report the failure within 5 seconds, naming the
  failing dependency, and answer only from data it actually retrieved —
  reporting the failure rather than substituting fabricated or stale values.
- **FR-007**: The system MUST decline questions outside Polymarket data and
  state its supported scope in the same answer.
- **FR-008**: When a question is ambiguous, the system MUST either ask one
  clarifying question or answer under a stated default assumption
  (24-hour volume for "top" queries), so every assumption is visible to the
  analyst.

### Non-Functional Requirements

- **NFR-001**: Single-market lookups complete within 3 seconds and multi-market
  scans over up to 1,000 markets within 10 seconds, at the 95th percentile,
  measured at the system boundary.
- **NFR-002**: The system supports at least 20 concurrent analyst sessions
  while still meeting NFR-001.
- **NFR-003**: Answers reflect Polymarket data no older than 60 seconds at the
  time of the query, and each answer states its data-as-of timestamp
  (see FR-005).

### Key Entities *(include if feature involves data)*

- **User Query**: A natural-language question from the analyst, arriving via
  their assistant; may name markets, metrics, categories, tags, date ranges, or
  open/closed status.
- **Polymarket Data**: Market, event, and trade records fetched from the
  Polymarket data source; each record carries the timestamp it was fetched.
- **Analysis Result**: The deterministic output of ranking, filtering, or
  summarizing Polymarket Data in response to a User Query.
- **Response**: The analyst-facing answer: prose plus, where tabular, a
  markdown table; always includes units and the data-as-of timestamp.

### Out of Scope

- A bespoke chat window, web UI, or command-line interface — the analyst's
  MCP-capable assistant is the interface (FR-001).
- Placing trades, portfolio management, or any write operation against
  Polymarket.
- Language understanding inside the system itself: interpreting the analyst's
  natural language is the assistant's job; the system exposes structured
  search and analysis capabilities.

What the eval taught the grader

Dogfooding cut both ways: grading a real repo exposed three linter bugs — the clarification-marker count includes the template's own instruction text (every fresh Spec-Kit spec starts ~4 markers in the hole), the acceptance-criteria check fails Spec-Kit's canonical sibling-heading layout, and the required-sections check demands template boilerplate the semantic judge simultaneously flags for removal.
